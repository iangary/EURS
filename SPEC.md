## 一、專案概述

### 1.1 背景與目標

工地／部門員工申請個人配備（安全帽、安全鞋、制服）目前以紙本與口頭通知為主，造成下列問題：

- 總務部彙整出貨費時
- 會計對帳困難
- 申請進度不透明

本系統建立一個集中化的線上申請平台，將「申請 → 出貨 → 對帳」三端流程數位化。

### 1.2 三端使用情境

| 端別     | 角色       | 主要任務                                            |
| -------- | ---------- | --------------------------------------------------- |
| 工地端   | 申請人     | 線上填寫表單送出申請，系統自動以 Email 通知總務     |
| 總務端   | 總務承辦   | 彙整申請資訊出貨、標記出貨狀態、處理退件            |
| 會計端   | 會計人員   | 透過匯出功能取得整合資料以利對帳                    |

### 1.3 預期效益

- 申請流程線上化，減少紙本與重複輸入
- Email 與系統雙重通知，避免漏單
- 申請人即時追蹤進度，減少詢問成本
- 總務出貨資料結構化，易於彙整與管理
- 會計可一鍵匯出指定區間資料，加速對帳

## 二、系統架構與角色權限

### 2.1 使用者角色

| 角色           | 登入方式                | 權限範圍                                                       |
| -------------- | ----------------------- | -------------------------------------------------------------- |
| 申請人         | 模擬登入（未來改 SSO）  | 建立申請、查詢自己的申請、追蹤進度、查詢自己的歷史紀錄         |
| 總務（管理員） | 模擬登入（未來改 SSO）  | 查詢／篩選所有申請、標記出貨、退件、匯出、維護系統參數         |
| 會計           | 不直接登入              | 由總務匯出 Excel／CSV 提供，作為對帳之用                       |

> 註：第一階段採模擬登入加角色切換器，介面預留 NextAuth Provider 介接點，未來可平順切換至公司 AD／SSO。

### 2.2 認證與授權

- 第一階段：模擬登入（從員工 API 撈取人員清單），首頁可切換角色測試
- 員工資料來源：`http://172.16.105.117/emp`（Swagger：`/emp/docs#/`）
- 角色判定：以系統參數 `ADMIN_EMPLOYEE_IDS` 白名單**或** `User.role = ADMIN` 為準（後台可維護）；`/api/employees/login-list` 依此回傳 `isAdmin` 旗標
- Session 逾時：閒置 30 分鐘自動登出
- 未來擴充：以 NextAuth 介接 Azure AD、反向代理 Header 或 LDAP

### 3.1 共通功能

- 登入／登出（模擬，介面預留 SSO Provider）
- 首頁：顯示「我的申請」近期 5 筆與狀態，加上三個快速申請大按鈕
- 導覽列：申請新單（安全帽／安全鞋／制服）、我的申請、（總務）後台管理、使用者選單

#### 3.4.3 業務邏輯

- **新領**：直接送出，無附加動作
- **更換**：強制要求上傳附件（舊衣照片），未上傳無法送出（前後端雙重驗證）
- **自購**：送出前跳出匯款資訊視窗，要求申請人確認；表單下方的黃色提示區也會直接顯示「匯款帳號：{`BANK_BRANCH`} {`BANK_ACCOUNT`}」，方便申請人事先預備
- 數量上限：每子項目 1～5；上衣與折褲分別計算
- 上衣與折褲的領用方式可不同（例：上衣選「新領」、折褲選「更換」）
- 若同一申請中**任一子項目**選「自購」，匯款視窗只彈出一次（統一顯示）
- 若同一申請中**任一子項目**選「更換」，附件區顯示為紅框並必填，附件可被多個「更換」子項目共用

#### 3.4.4 自購提示視窗

| 項目          | 內容                                      |
| ------------- | ----------------------------------------- |
| 說明          | 請與總務聯繫取得購買金額                  |
| 匯款銀行／分行| 中崙分行                                  |
| 匯款帳號      | 045-031-0000-1898                         |
| 按鈕          | 「我已知悉並確認送出」／「取消」          |

> 實作上匯款資訊改由系統參數（`SystemSetting`）後台維護，避免日後變更時須重新部署。

#### 3.4.5 附件規格

- 允許格式：JPG／PNG／PDF
- 單檔大小：不超過 10 MB
- 每筆申請：附件數不超過 5 個

### 3.5 批量匯入（Excel）

為減少人工逐列輸入，三類申請皆提供批量匯入功能。流程：**下載範本 → 填寫 Excel → 上傳 → 系統解析並預覽 → 確認送出**。

#### 3.5.1 範本下載

- 入口：每個申請頁面右上角「下載範本」按鈕；以及共用頁面 `/templates`
- 範本檔名：
  - `EURS_批量範本_安全帽.xlsx`
  - `EURS_批量範本_安全鞋.xlsx`
  - `EURS_批量範本_制服.xlsx`
- 範本內容：
  - 第 1 列：欄位名稱（必填欄以 `*` 標示）
  - 第 2 列：欄位說明與範例值（淺灰底，匯入時會被略過）
  - 第 3 列起：使用者填寫資料
  - 工作表 `說明`：填寫規則、選項列表（血型、鞋號、性別、品項、類型）
  - 已預設「資料驗證」下拉，避免拼錯（例如 鞋號限 36–46）

#### 3.5.2 各類別範本欄位

| 類別     | 必填欄位                                                                  | 選填欄位 |
| -------- | ------------------------------------------------------------------------- | -------- |
| 安全帽   | 使用人姓名、血型                                                          | 備註     |
| 安全鞋   | 使用人姓名、鞋號、說明原因                                                | 備註     |
| 制服     | 使用人姓名、性別、（上衣）尺寸／件數／領用方式、（折褲）腰圍／褲長／件數／領用方式 — 上衣與折褲至少填一組 | 備註     |

> 「申請人」與「所屬工地／部門」由登入者自動帶入，**不出現在範本中**，每次匯入屬於一張申請單。

#### 3.5.3 上傳與解析

- 接受副檔名：`.xlsx`（不接受 `.xls`，避免相容性問題）
- 單檔大小：不超過 5 MB
- 單次匯入列數：上限 200 列（避免逾時；超過建議拆檔）
- 後端使用 `exceljs` 解析 → 進行欄位驗證與業務規則驗證
- 解析結果以「預覽表」呈現，每列顯示三種狀態：
  - ✅ 通過：資料合法，可送出
  - ⚠️ 警告：可送出但建議檢查（例如備註過長已截斷）
  - ❌ 錯誤：紅底標記，列出原因（必填缺漏、選項不在白名單、件數超過上限、制服「更換」缺附件等）
- 錯誤列阻擋整批送出；使用者可選擇「修正後重新上傳」或「僅送出通過列」（後者預設關閉，避免誤送）

#### 3.5.4 制服類別特別規則

- 制服類型為「更換」時，附件無法在 Excel 中表達；解決方式：
  - 在預覽頁針對每一列「更換」列提供獨立的「上傳附件」欄
  - 每列至少上傳一個附件才能送出，否則該列標記為錯誤
- 自購匯款提示視窗在預覽頁送出前統一顯示一次（若批次包含任何「自購」列）

#### 3.5.5 錯誤訊息規格（範例）

| 錯誤代碼          | 訊息                                            |
| ----------------- | ----------------------------------------------- |
| `MISSING_FIELD`   | 第 5 列「使用人姓名」為必填                     |
| `INVALID_OPTION`  | 第 7 列「血型」必須為 A／B／O／AB              |
| `OUT_OF_RANGE`    | 第 9 列「鞋號」需介於 36 與 46 之間             |
| `OVER_LIMIT`      | 第 12 列「上衣件數」或「折褲件數」需介於 1～5   |
| `MISSING_ATTACH`  | 第 14 列為「更換」，請於預覽頁補上附件          |
| `TOO_MANY_ROWS`   | 一次最多匯入 200 列，目前 215 列                |

#### 3.5.6 匯入結果

- 一次匯入產生**一張**申請單，包含多筆 `RequestItem`
- 申請單號規則同手動建單（`PPE-YYYYMMDD-NNN`）
- 通知機制同 §3.9（一次匯入只發一封 Email 給總務，附件數摘要列在信中）
- 匯入歷程於 `StatusLog` 留下 `note: 由 Excel 匯入 N 列`

### 3.6 我的申請與進度追蹤

- 查詢「自己提出」的所有申請（含歷史）
- 清單欄位：申請單號、申請日期、項目類型、品項摘要、目前狀態、出貨日期
- 狀態類型：`已送出（待處理）`、`已出貨`、`退件`
- 篩選條件：項目類型、狀態、申請日期區間
- 點選單筆可查看完整內容與狀態變更紀錄（`StatusLog`）
- 狀態為 `退件` 的申請，申請人可在詳情頁點「重新填寫並送出」，系統以 `?from=<requestId>` 帶入原始 `dept`／`remark`／使用人明細到對應申請表單預填，送出後產生新單號（保留原退件單稽核），對應 §10 第 10 項預設方案

### 3.7 總務後台管理

#### 3.7.1 申請單管理

- 單一表格清單，無分頁籤；表頭下方一列內嵌篩選欄，前端即時套用：
  - 申請單號（文字搜尋）
  - 項目類型（下拉：全部／安全帽／安全鞋／制服）
  - 申請人姓名（文字搜尋）
  - 工地／部門（文字搜尋）
  - 使用人姓名（文字搜尋，比對該申請所有 `RequestItem.userName`）
  - 狀態（下拉：全部／已送出／已出貨／退件）
- 上方卡片提供日期區間（起／迄）伺服器端篩選與「批次標記已出貨」按鈕
- 進階篩選（後續擴充）：（安全鞋）鞋號、（制服）子項目、（制服）領用方式、（制服）性別
- 排序：申請日期、狀態

#### 3.7.2 出貨標記

- 單筆「標記已出貨」：跳出確認框，確認後狀態改為「已出貨」並紀錄出貨日期與操作人員
- 已出貨可「還原」（狀態回到 `已送出`，必留稽核紀錄）；對應 API：`POST /api/requests/{id}/unship`，會清空 `shippedAt` / `shippedById` 並寫一筆 `StatusLog`
- 批次勾選後可一鍵標記已出貨（第二期）

#### 3.7.3 退件處理

- 改為「退件」狀態，須填寫退件原因
- 退件後系統自動以 Email 通知申請人（內含原因）

### 3.8 匯出功能（會計對帳）

- 入口：總務後台 → 匯出
- 日期欄位採用萬年曆（date picker），不再使用下拉年月
- 匯出條件：項目類型、出貨日期區間、狀態、工地／部門；出貨日期區間預設「近 7 天」，並可勾選「包含尚未出貨」
- 匯出格式：Excel（.xlsx）為主，可選 CSV
- 「預覽」按鈕：呼叫 `GET /api/admin/export?format=preview`，回傳 JSON `{ header, rows, count }`，於頁面內以表格顯示，確認資料後再下載 Excel／CSV
- 匯出欄位：申請單號、申請日期、申請人、工地／部門、項目類型、使用人、規格（血型／鞋號／上衣尺寸／腰圍／褲長）、子項目（上衣／折褲）、數量、領用方式（新領／更換／自購）、狀態、出貨日期、備註
- 檔名範例：`PPE_出貨明細_2026-01-01_2026-03-31.xlsx`
- 5,000 筆以內同步下載；超過則背景處理並以 Email 寄送下載連結

### 3.9 通知機制

| 觸發時機                     | 通知對象          | 通知方式                                  |
| ---------------------------- | ----------------- | ----------------------------------------- |
| 申請人送出申請               | 總務（全部承辦）  | Email 加系統（後台清單即時更新）          |
| 總務標記已出貨               | 申請人            | Email 加系統（我的申請更新狀態）          |
| 申請被退件                   | 申請人            | Email 加系統（含退件原因）                |

Email 內容建議包含：申請單號、項目摘要、目前狀態、後台深層連結（可直接跳到該筆申請）。

## 六、介面說明

### 6.1 登入後首頁
- 頂部導覽列：商標、申請項目選單、我的申請、（總務）申請單管理、使用者名稱
- 主區塊：我的申請快速清單（最近 5 筆）與「立即申請」大按鈕
- 登入頁快速登入清單：管理員（`isAdmin = true`）姓名旁顯示「管理員」徽章，方便切換測試

### 6.2 安全帽申請畫面
- 申請人與所屬工地以「自動帶入」區塊呈現
- 使用人姓名與血型成對輸入，「＋新增欄位」可增列
- 數量固定為 1，顯示為唯讀
- 底部「送出申請」主按鈕

### 6.3 安全鞋申請畫面
- 「鞋號」由文字輸入改為下拉選單（36～46）
- 「說明原因」為必填多行文字

### 6.4 制服申請畫面
- **卡片化階層**：主分類「制服」用較深底色框住整張卡片，子項目「上衣」與「折褲」用淺灰底加細邊框，視覺層級一眼可辨
- **基本資料區**：申請人、所屬工地、使用人姓名（可動態增列）、性別（男／女款）— 性別位於主分類層級，整張卡片只選一次
- **上衣子卡**：勾選啟用後展開內容；尺寸採按鈕式單選（S／M／L／XL／2XL／3XL），點按即選，行動裝置友善；數量數字輸入；領用方式為三選一 Radio（新領／更換／自購）
- **折褲子卡**：勾選啟用後展開內容；腰圍與褲長為兩個下拉選單；數量數字輸入；領用方式同上
- **黃色提示區**：只在必要時出現
  - 任一子項目選「更換」 → 顯示附件上傳區並標紅框必填
  - 任一子項目選「自購」 → 送出前彈出匯款資訊視窗（同一申請只彈一次）
- 上衣與折褲必須至少勾選其中一項，否則「送出」按鈕禁用

### 6.5 批量匯入畫面
- 上方提示文字加「下載範本」按鈕（依目前申請類別下載對應範本）
- 中間檔案上傳區，支援拖放與點選
- 上傳後顯示預覽表，每列以 ✅ ⚠️ ❌ 標示狀態
- 錯誤列右側顯示原因；制服「更換」列另有附件上傳欄
- 底部「送出全部」按鈕（有 ❌ 時禁用）與「重新上傳」按鈕

### 6.6 後台清單畫面
- 單一表格、無分頁籤；表頭下方一列為各欄位篩選輸入（文字搜尋／下拉），前端即時過濾
- 上方卡片提供日期區間（伺服器端篩選）與「批次標記已出貨」按鈕
- 每列右側「已出貨」按鈕，標記後變綠色「已出貨 ✓」；已出貨可「還原」

### 6.7 匯出畫面
- 日期區間採用萬年曆（date picker），出貨日期預設「近 7 天」
- 項目類型下拉、狀態下拉、工地／部門關鍵字、「包含尚未出貨」核取
- 三顆按鈕：「預覽」（頁內顯示結果表）、「匯出 CSV」、「匯出 Excel」

## 七、非功能性需求

### 7.1 安全性
- 一律走 HTTPS（正式環境）
- 身分驗證採 SSO（第二期），系統不另存密碼
- 檔案上傳檢查副檔名與 MIME type，並限制檔案大小
- 敏感操作（出貨、退件、匯出）留稽核紀錄
- 附件下載須通過權限檢查，不可直接公開存取卷

### 7.2 效能
- 一般頁面回應時間少於 2 秒
- 匯出 5,000 筆以內須在 10 秒內完成；超過則改為背景處理並以 Email 寄送下載連結

### 7.3 可用性
- 支援 Chrome、Edge、Safari 最新兩個主要版本
- 支援 RWD，工地端可用平板或手機操作
- 介面以繁體中文為主

### 7.4 維運
- 申請資料保留至少 5 年（配合會計帳務）
- 附件儲存採用 Docker 命名卷，定期備份
- 系統參數（匯款帳號、鞋號、血型、總務名單）由後台維護，免改程式

## 十、待確認事項

> 以下事項建議在開發啟動前由總務部與相關單位確認；本系統先採「預設方案」實作，可隨時調整。

1. 血型選項是否含「不詳」、「不提供」？ **預設 A／B／O／AB（系統參數可擴充）**
2. 安全鞋鞋號是否需支援半號（如 41.5）？ **預設整數 36～46**
3. 制服「自購」匯款後回單是否需上傳系統？ **預設不需要（可加為新附件類別）**
4. 總務角色是否要再分「主管」與「承辦」？ **預設單一管理員**
5. 是否對「使用人」建立員工主檔？ **預設自由文字，選用 /emp 自動完成**
6. 申請是否限定本部門員工？ **預設不限制（可代他部門申請）**
7. 制服自購匯款帳號是否會多帳號？ **預設單組，多組改為 keyed map**
8. Email 通知是否支援多收件人或副本？ **支援，由 `ADMIN_NOTIFY_EMAILS` 設定**
9. 會計匯出是否有指定範本？ **先依 §3.8 欄位實作，UAT 後再微調**
10. 退件後申請人如何重送？ **預設由系統將退件單複製為新申請（新單號，保留稽核）**（已實作為 §3.6「重新填寫並送出」流程）
