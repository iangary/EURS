## 一、專案概述

### 1.1 背景與目標

工地／部門員工申請個人配備（安全帽、安全鞋、制服）目前以紙本與口頭通知為主，造成下列問題：

- 總務部彙整出貨費時
- 會計對帳困難
- 申請進度不透明

本系統建立一個集中化的線上申請平台，將「申請 → 出貨 → 對帳」三端流程數位化。

### 1.2 三端使用情境

| 端別     | 角色       | 主要任務                                            |
| -------- | ---------- | --------------------------------------------------- |
| 工地端   | 申請人     | 線上填寫表單送出申請，系統自動以 Email 通知總務     |
| 總務端   | 總務承辦   | 彙整申請資訊出貨、標記出貨狀態、處理退件            |
| 會計端   | 會計人員   | 透過匯出功能取得整合資料以利對帳                    |

### 1.3 預期效益

- 申請流程線上化，減少紙本與重複輸入
- Email 與系統雙重通知，避免漏單
- 申請人即時追蹤進度，減少詢問成本
- 總務出貨資料結構化，易於彙整與管理
- 會計可一鍵匯出指定區間資料，加速對帳

## 二、系統架構與角色權限

### 2.1 使用者角色

| 角色           | 登入方式                | 權限範圍                                                       |
| -------------- | ----------------------- | -------------------------------------------------------------- |
| 申請人         | 模擬登入（未來改 SSO）  | 建立申請、查詢自己的申請、追蹤進度、查詢自己的歷史紀錄         |
| 總務（管理員） | 模擬登入（未來改 SSO）  | 查詢／篩選所有申請、標記出貨、退件、匯出、維護系統參數         |
| 會計           | 不直接登入              | 由總務匯出 Excel／CSV 提供，作為對帳之用                       |

> 註：第一階段採模擬登入加角色切換器，介面預留 NextAuth Provider 介接點，未來可平順切換至公司 AD／SSO。

### 2.2 認證與授權

- 第一階段：模擬登入（從員工 API 撈取人員清單），首頁可切換角色測試
- 員工資料來源：`http://172.16.105.117/emp`（Swagger：`/emp/docs#/`）
- 角色判定：以系統參數 `ADMIN_EMPLOYEE_IDS` 白名單為準（後台可維護）
- Session 逾時：閒置 30 分鐘自動登出
- 未來擴充：以 NextAuth 介接 Azure AD、反向代理 Header 或 LDAP

### 2.3 系統高層架構

```
┌──────────────┐    ┌────────────────────────────────────┐
│   瀏覽器     │ ←→ │ Next.js (App Router)（部署於容器） │
│ 工地／總務   │    │  ├─ React 介面（繁體中文）         │
└──────────────┘    │  ├─ Server Actions／API Routes     │
                    │  ├─ NextAuth（模擬 Provider）      │
                    │  └─ Prisma Client                  │
                    └───────┬────────────────────┬───────┘
                            │                    │
                ┌───────────▼─────┐   ┌──────────▼──────────┐
                │  PostgreSQL 16  │   │   外部 HTTP API     │
                │ （Docker 卷）   │   │  ├─ /emp 員工查詢   │
                └─────────────────┘   │  └─ /app 寄信服務   │
                                      └─────────────────────┘
                ┌─────────────────┐
                │ uploads 卷      │ ← 制服「更換」附件
                └─────────────────┘
```

## 三、功能需求

### 3.1 共通功能

- 登入／登出（模擬，介面預留 SSO Provider）
- 首頁：顯示「我的申請」近期 5 筆與狀態，加上三個快速申請大按鈕
- 導覽列：申請新單（安全帽／安全鞋／制服）、我的申請、（總務）後台管理、使用者選單
- 所有申請表單支援「＋新增欄位」按鈕，可動態新增「使用人」列
- 使用者可隨時查詢自己申請的歷史紀錄與目前進度狀態

### 3.2 安全帽申請

#### 3.2.1 表單欄位

| 欄位            | 型態      | 必填 | 說明                              |
| --------------- | --------- | :--: | --------------------------------- |
| 申請人          | 自動帶入  |  是  | 由登入帳號帶入，不可修改          |
| 所屬工地／部門  | 下拉選單  |  是  | 由員工 API 帶入，可手動切換       |
| 使用人姓名      | 文字      |  是  | 可動態新增多列                    |
| 血型            | 下拉選單  |  是  | A／B／O／AB（可由系統參數維護） |
| 數量            | 固定值    |  ─   | 每筆固定為 1                      |
| 備註            | 多行文字  |  否  | 申請人補充說明                    |

#### 3.2.2 操作行為

- 「＋新增欄位」可動態新增「使用人姓名與血型」一列
- 送出前進行必填驗證；通過後寫入資料庫並產生申請單號
- 送出成功後：系統發送 Email 通知總務，並於前台顯示成功訊息

### 3.3 安全鞋申請

#### 3.3.1 表單欄位

| 欄位            | 型態      | 必填 | 說明                                   |
| --------------- | --------- | :--: | -------------------------------------- |
| 申請人          | 自動帶入  |  是  |                                        |
| 所屬工地／部門  | 下拉選單  |  是  |                                        |
| 使用人姓名      | 文字      |  是  | 可動態新增多列                         |
| 鞋號            | 下拉選單  |  是  | 36～46（可由系統參數維護）            |
| 說明原因        | 多行文字  |  是  | 工地申請理由                           |
| 數量            | 固定值    |  ─   | 每筆固定為 1                           |

#### 3.3.2 業務規則：90 天領取期限

- 同一使用人取得安全鞋後 **90 天內不得再次申請**
- 系統自動檢核並擋件，回應訊息：`該使用人尚在 90 天領取期限內`
- 90 天起算日：以該使用人「已出貨」之日期為準
- 特殊情況可由總務後台覆蓋（必留稽核紀錄於 `StatusLog`）

### 3.4 制服申請

#### 3.4.1 表單欄位

| 欄位            | 型態          | 必填   | 說明                                |
| --------------- | ------------- | :----: | ----------------------------------- |
| 申請人          | 自動帶入      |  是    |                                     |
| 所屬工地／部門  | 下拉選單      |  是    |                                     |
| 使用人姓名      | 文字          |  是    | 可動態新增多列                      |
| 性別            | 單選（男／女）|  是    | 影響制服類型                        |
| 品項            | 勾選          |  是    | 上衣／折褲（可複選）               |
| 類型            | 單選          |  是    | 更換／自購（觸發不同行為）         |
| 數量            | 數字          |  是    | 上限 5 件                           |
| 附件            | 檔案上傳      | 視類型 | 「更換」必填（舊衣照片）            |

#### 3.4.2 業務邏輯

- **更換**：強制要求上傳附件，未上傳無法送出（前後端雙重驗證）
- **自購**：送出前跳出提示視窗顯示匯款資訊，要求申請人確認
- 數量上限 5 件；上衣與折褲可分別填寫不同數量

#### 3.4.3 自購提示視窗

| 項目          | 內容                                      |
| ------------- | ----------------------------------------- |
| 說明          | 請與總務聯繫取得購買金額                  |
| 匯款銀行／分行| 中崙分行                                  |
| 匯款帳號      | 045-031-0000-1898                         |
| 按鈕          | 「我已知悉並確認送出」／「取消」          |

> 實作上匯款資訊改由系統參數（`SystemSetting`）後台維護，避免日後變更時須重新部署。

#### 3.4.4 附件規格

- 允許格式：JPG／PNG／PDF
- 單檔大小：不超過 10 MB
- 每筆申請：附件數不超過 5 個

### 3.5 我的申請與進度追蹤

- 查詢「自己提出」的所有申請（含歷史）
- 清單欄位：申請單號、申請日期、項目類型、品項摘要、目前狀態、出貨日期
- 狀態類型：`已送出（待處理）`、`處理中`、`已出貨`、`退件`
- 篩選條件：項目類型、狀態、申請日期區間
- 點選單筆可查看完整內容與狀態變更紀錄（`StatusLog`）

### 3.6 總務後台管理

#### 3.6.1 申請單管理

- 分頁籤：`全部`、`安全帽`、`安全鞋`、`制服`
- 篩選條件：申請日期區間、申請人、工地／部門、使用人、狀態、（安全鞋）鞋號、（制服）類型
- 關鍵字搜尋：申請單號、姓名
- 排序：申請日期、狀態

#### 3.6.2 出貨標記

- 單筆「標記已出貨」：跳出確認框，確認後狀態改為「已出貨」並紀錄出貨日期與操作人員
- 已出貨可「還原」（必留稽核紀錄）
- 批次勾選後可一鍵標記已出貨（第二期）

#### 3.6.3 退件處理

- 改為「退件」狀態，須填寫退件原因
- 退件後系統自動以 Email 通知申請人（內含原因）

### 3.7 匯出功能（會計對帳）

- 入口：總務後台 → 匯出
- 日期欄位採用萬年曆（date picker），不再使用下拉年月
- 匯出條件：項目類型、申請日期區間、出貨日期區間、狀態、工地／部門
- 匯出格式：Excel（.xlsx）為主，可選 CSV
- 匯出欄位：申請單號、申請日期、申請人、工地／部門、項目類型、使用人、規格（血型／鞋號／品項）、數量、類型（制服）、狀態、出貨日期、備註
- 檔名範例：`PPE_出貨明細_2026-01-01_2026-03-31.xlsx`
- 5,000 筆以內同步下載；超過則背景處理並以 Email 寄送下載連結

### 3.8 通知機制

| 觸發時機                     | 通知對象          | 通知方式                                  |
| ---------------------------- | ----------------- | ----------------------------------------- |
| 申請人送出申請               | 總務（全部承辦）  | Email 加系統（後台清單即時更新）          |
| 總務標記已出貨               | 申請人            | Email 加系統（我的申請更新狀態）          |
| 申請被退件                   | 申請人            | Email 加系統（含退件原因）                |
| 安全鞋 90 天期限到期前 7 天  | 申請人（可選）    | Email 提醒可重新申請                      |

Email 內容建議包含：申請單號、項目摘要、目前狀態、後台深層連結（可直接跳到該筆申請）。

## 四、資料模型

> 採 Prisma 加 PostgreSQL，完整 schema 見 `prisma/schema.prisma`。

### 4.1 列舉型別

```prisma
enum RequestType   { HELMET SHOES UNIFORM }                       // 安全帽／安全鞋／制服
enum RequestStatus { SUBMITTED PROCESSING SHIPPED REJECTED }      // 已送出／處理中／已出貨／退件
enum UniformAction { REPLACE PURCHASE }                           // 更換／自購
enum Gender        { MALE FEMALE }                                // 男／女
enum BloodType     { A B O AB }                                   // 血型
enum Role          { REQUESTER ADMIN }                            // 申請人／總務
```

## 七、非功能性需求

### 7.1 安全性
- 一律走 HTTPS（正式環境）
- 身分驗證採 SSO（第二期），系統不另存密碼
- 檔案上傳檢查副檔名與 MIME type，並限制檔案大小
- 敏感操作（出貨、退件、匯出、覆蓋 90 天）留稽核紀錄
- 附件下載須通過權限檢查，不可直接公開存取卷

### 7.2 效能
- 一般頁面回應時間少於 2 秒
- 匯出 5,000 筆以內須在 10 秒內完成；超過則改為背景處理並以 Email 寄送下載連結

### 7.3 可用性
- 支援 Chrome、Edge、Safari 最新兩個主要版本
- 支援 RWD，工地端可用平板或手機操作
- 介面以繁體中文為主

### 7.4 維運
- 申請資料保留至少 5 年（配合會計帳務）
- 附件儲存採用 Docker 命名卷，定期備份
- 系統參數（匯款帳號、鞋號、血型、總務名單）由後台維護，免改程式

## 十、待確認事項

> 以下事項建議在開發啟動前由總務部與相關單位確認；本系統先採「預設方案」實作，可隨時調整。

1. 血型選項是否含「不詳」、「不提供」？ **預設 A／B／O／AB（系統參數可擴充）**
2. 安全鞋鞋號是否需支援半號（如 41.5）？ **預設整數 36～46**
3. 制服「自購」匯款後回單是否需上傳系統？ **預設不需要（可加為新附件類別）**
4. 總務角色是否要再分「主管」與「承辦」？ **預設單一管理員，覆蓋 90 天需管理員權限**
5. 是否對「使用人」建立員工主檔？ **預設自由文字，選用 /emp 自動完成**
6. 申請是否限定本部門員工？ **預設不限制（可代他部門申請）**
7. 制服自購匯款帳號是否會多帳號？ **預設單組，多組改為 keyed map**
8. Email 通知是否支援多收件人或副本？ **支援，由 `ADMIN_NOTIFY_EMAILS` 設定**
9. 會計匯出是否有指定範本？ **先依 §3.7 欄位實作，UAT 後再微調**
10. 退件後申請人如何重送？ **預設由系統將退件單複製為新申請（新單號，保留稽核）**
