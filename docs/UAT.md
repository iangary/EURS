## 四、制服申請（SPEC §3.4, §6.4）

路由：`/apply/uniform`

### 4.1 基本驗證

- 上衣與折褲皆未勾選 → 「送出」按鈕禁用
- 性別位於主分類層級，整張卡片只選一次

### 4.2 上衣子卡

- 勾選啟用 → 展開內容
- 尺寸採按鈕式單選（S／M／L／XL／2XL／3XL），點按即選
- 數量限制 1–5（輸入 0 或 6 應被阻擋）
- 領用方式三選一互斥

### 4.3 折褲子卡

- 腰圍／褲長為下拉，選項由 `PANTS_WAIST` / `PANTS_LENGTH` 提供

### 4.4 三種領用方式分支

| 分支 | 步驟                              | 預期                                                                                   |
| ---- | --------------------------------- | -------------------------------------------------------------------------------------- |
| 新領 | 上衣選新領、送出                  | 直接送出，無附件、無視窗                                                               |
| 更換 | 折褲選更換、不上傳 → 送出         | 阻擋送出，附件區紅框；上傳照片後可送出                                                 |
| 自購 | 上衣選自購 → 送出                 | 跳出匯款視窗，顯示 `BANK_BRANCH` `BANK_ACCOUNT`；點「我已知悉並確認送出」始送出       |
| 混合 | 上衣＝更換、折褲＝自購            | 附件必填＋匯款視窗只彈一次                                                             |

### 4.5 附件規格

- 上傳 11 MB 檔 → 阻擋
- 上傳 .exe → 阻擋（副檔名／MIME 雙重檢查）
- 第 6 個附件 → 阻擋（每單上限 5）

## 七、總務後台（SPEC §3.7, §6.6）

需以 `A001` 或 `RT024898` 登入。

### 7.1 申請單管理（`/admin/requests`）

- 表頭內嵌篩選欄：申請單號、項目類型、申請人、工地、使用人姓名、狀態 — 前端即時過濾
- 上方卡片：日期區間（伺服器端）、批次標記已出貨按鈕
- 排序：申請日期、狀態

### 7.2 出貨標記

| #     | 步驟                                | 預期                                                                                       |
| ----- | ----------------------------------- | ------------------------------------------------------------------------------------------ |
| 7.2.1 | 點單列「已出貨」 → 確認框 → 確認    | 狀態變綠色「已出貨 ✓」；`shippedAt`／`shippedById` 寫入；申請人收到 Email                  |
| 7.2.2 | 已出貨單點「還原」                  | `POST /api/requests/{id}/unship`；`shippedAt`／`shippedById` 清空；`StatusLog` 留紀錄      |
| 7.2.3 | 勾多筆 → 批次標記已出貨             | 全部更新為已出貨，個別寫入 `StatusLog`                                                     |
| 7.2.4 | 嘗試對已出貨單退件                  | 阻擋                                                                                       |

### 7.3 退件

- 點退件 → 必填退件原因 → 確認 → 狀態變「退件」；申請人收到含原因的 Email

## 九、通知機制（SPEC §3.9）

每段流程後檢查 mail-api log（或 mail server）：

| 觸發            | 收件人                    | 內容                       |
| --------------- | ------------------------- | -------------------------- |
| 申請送出        | `ADMIN_NOTIFY_EMAILS`     | 單號＋摘要＋後台連結       |
| 標記已出貨      | 申請人                    | 狀態更新                   |
| 退件            | 申請人                    | 含退件原因                 |

連線失敗時：申請流程不中斷，僅 log 寫錯誤（驗證 mail-api 服務暫停的情境）。

## 十、系統參數（SPEC §4 SystemSetting）

路由：`/admin/settings`

- 修改 `BANK_BRANCH`／`BANK_ACCOUNT` → 制服自購視窗即時反映
- 修改 `SHOE_SIZES` → 安全鞋下拉同步
- 修改 `BLOOD_TYPES`／`TOP_SIZES`／`PANTS_WAIST`／`PANTS_LENGTH` → 對應表單同步
- 修改 `ADMIN_NOTIFY_EMAILS`／`ADMIN_EMPLOYEE_IDS` → 送出後立即生效
